**Constraints**(额外):

- 两层接同一签名:`task` `prompt-path` `context`(JSON)`model` `max-turns`,outputs `result` `comment-id`
- prompt 加载优先级:`prompt-path` 非空 → 调用方仓库读;空 → `prompts/{task}.md` 内置
- 必须 `use_sticky_comment: true`(同 PR 多次触发只更新同一条评论)

**Implementation outline**:
1. 先写 composite `_common/claude-harness/action.yml`,内部 `uses: anthropics/claude-code-action@<SHA>`
2. composite 处理 prompt 加载:`if [ -n "$PROMPT_PATH" ]; then cat $PROMPT_PATH; else cat ${{ github.action_path }}/../../../prompts/$TASK.md; fi`
3. composite emit annotation
4. 写 reusable workflow `claude-harness.yml`,内部一个 job `ai-task`,挂 `_common/claude-harness` composite
5. 把内置 prompt 各写一版(具体 prompt 内容由后续 P1/P2 任务补,本任务先放占位)

**Acceptance criteria**:

- [ ] PR 触发后 GitHub PR 上**只有一条** AI review 评论(sticky),后续 push 更新同一条
- [ ] 不传 `prompt-path` → 用 `prompts/pr/review.md`
- [ ] 传 `pr-review-prompt-path: .agents/skills/custom.md`(消费方仓库相对路径) → 该路径被读取
- [ ] 缺 `ANTHROPIC_API_KEY` → preflight fail,annotation `::error title=Missing Secret::ANTHROPIC_API_KEY`
- [ ] 通过 reusable workflow 调用(`uses: ./.github/workflows/claude-harness.yml`):job 完整跑通,outputs.result 非空
- [ ] 通过 composite 调用(在另一原子内部 `uses: ./actions/_common/claude-harness`):同样 outputs.result 非空

**Constraints**(额外):

5. `ci.yml` 串接,`build-docker` outputs 通过 `needs.build-docker.outputs.digest` 传给后续

**Acceptance criteria**:
- [ ] 在 e2e 仓库(P1-11 同步建立)merge to main → ci.yml 触发,success
- [ ] `ghcr.io/<owner>/<image-name>:main` tag 可拉取
- [ ] `ghcr.io/<owner>/<image-name>:sha-abc1234` tag 可拉取(短 7 位)
- [ ] `cosign verify ghcr.io/<owner>/<image-name>@sha256:<digest>` 成功(keyless,GitHub OIDC issuer)
- [ ] Trivy 扫描结果在 GitHub repo Security tab 可见
- [ ] 故意引入高危 CVE 依赖(如 `lodash@4.17.10`) → `scan-image` job fail
- [ ] outputs `image-digest` 在调用方 workflow 通过 `needs.build.outputs.image-digest` 可读
- [ ] 同时连续 push 两次到 main:两个 ci.yml run 都跑完(不取消)
- [ ] `enable-ai-smoke: true` + 配置 `ANTHROPIC_API_KEY` → eval-smoke job 跑通,sticky comment 出现在最近一次相关 PR 上(若存在)

## P1-12: 观察窗口迁移到 repository_dispatch

**Status**: 🔴 Not Started

**Goal**:把 `actions/prd/observe-window` 的 `sleep` 实现迁移到 `repository_dispatch` 延迟触发模式,不再占 runner 分钟。

**SPEC ref**:5.5 已知技术债

**Effort**:M(3 天,含外部调度器 / GitHub Scheduled Workflows 的桥接)

**Dependencies**:P1-11(stg.yml 需要回写 `stg-deploy-time` 到 GitHub Variable),P1-13(prd.yml 须接 `repository_dispatch` 触发器)

**Files to create / modify**:
- `actions/prd/observe-window/action.yml`(改为纯校验,**不 sleep**)
- `actions/_common/schedule-prd-dispatch/action.yml`(新原子:在 stg.yml 末尾调用,设置在 `deploy-time + observation-minutes` 时刻发送 repository_dispatch)
- `.github/workflows/stg.yml`(P1-11 文件,新增 `schedule-prd-dispatch` step)
- `.github/workflows/prd.yml`(P1-13 文件,新增 `on: repository_dispatch: types: [observe-window-complete]` 触发器)

**Constraints**(额外):
- 旧的 `sleep $WAIT` 实现**必须删除**,observe-window 退化为纯校验:验证 `elapsed >= required`,否则 `::error`(意味着 prd 被错误地早触发)
- 调度方案选 A(SPEC 5.5):用 `gh api repos/.../dispatches` 配合外部 cron(可用 GitHub Actions 自身的 `schedule:` 在另一 polling workflow 中实现,或外部如 GitHub Scheduled Workflows)
- repository_dispatch payload 必须含:`stg-deploy-time` `image-digest` `stg-image-digest` 三个字段
- prd.yml 接到 dispatch 后,从 payload 读上述字段并触发完整 prd 流程
- 迁移期间 P1-13 临时实现可保留 `sleep` 兜底(`if: github.event_name == 'workflow_dispatch'`),本任务完成后切换为 dispatch-driven

**Implementation outline**:
1. observe-window 重写为纯校验:`STG_TIME = inputs.stg-deploy-time`,`NOW = date +%s`,`ELAPSED = (NOW - STG_TIME) / 60`,`if [ $ELAPSED -lt $REQUIRED ]; then echo "::error"; exit 1; fi`
2. `schedule-prd-dispatch`:输入 `stg-deploy-time` `observation-minutes` `image-digest`,实现:
   - 计算 `dispatch-at = stg-deploy-time + observation-minutes`
   - 写入 GitHub Variable `PENDING_PRD_DISPATCHES`(JSON 数组)
   - 由独立 polling workflow(`poll-prd-dispatch.yml`,5 分钟 cron)读这个 variable,到点的发 `repository_dispatch`
3. stg.yml 末尾加一步调用 `schedule-prd-dispatch`
4. prd.yml 加 `on: repository_dispatch: types: [observe-window-complete]`,从 `github.event.client_payload.*` 读 inputs

**Acceptance criteria**:
- [ ] e2e 仓库测试:stg.yml 完成后,`gh variable get PENDING_PRD_DISPATCHES` 含一条记录
- [ ] 等待观察窗口结束(测试用 1 分钟而非 30 分钟):prd.yml 自动被 repository_dispatch 触发,无需手工
- [ ] prd.yml 收到的 inputs 与 stg.yml 输出的 deploy-time / digest 一致
- [ ] 如果手动用 `workflow_dispatch` 提前触发 prd.yml(早于窗口),observe-window job emit `::error title=Observation Window Too Early` + exit 1
- [ ] runner 时间测量:stg.yml 末尾调度 step < 30s(不再 sleep 30min)
- [ ] 月度 runner 时间统计:观察窗口造成的 runner 占用 → 0(对比 P1 早期实现节省 ~15h/月)

## P1-14: 部署回滚机制

**Status**: 🔴 Not Started(部分作为 P1-13 的 acceptance 已覆盖,本任务做强化)

**Goal**:把 P1-13 中的简易回滚强化为标准 composite,包括快照、回滚、incident issue 自动化。

**SPEC ref**:5.5(回滚策略段)

**Effort**:S(1 天,本质是把 P1-13 的 if-failure step 抽成独立 composite)

**Dependencies**:P1-13

**Files to create**:
- `actions/prd/auto-rollback/action.yml`(composite:snapshot + on-failure undo + incident issue)
- 修改 `actions/prd/smoke-test/action.yml`:失败时 emit signal 给 auto-rollback
- 修改 `.github/workflows/prd.yml`:smoke-test 后挂 `auto-rollback` job(`if: failure() && needs.smoke-test.result == 'failure'`)

**Constraints**(额外):
- 回滚操作必须**幂等**:多次跑 `kubectl rollout undo` 不应破坏更早的版本
- incident issue 必须含:故障 SHA、回滚到的 SHA、smoke-test 日志摘要(annotation 即可)
- incident issue label 严格 `incident,priority:p1`(不接受 `incident-p1` 等变体,因为后续 ops 工作流按这个 label 触发)
- auto-rollback 自身**不**应 fail prd workflow(`continue-on-error: true` 在 issue 创建步)— 回滚成功是首要目的,issue 创建是辅助

**Implementation outline**:
1. 抽 P1-13 的 if-failure step 到 `auto-rollback` composite
2. 加 snapshot step(`pre-deploy` 时记录 `kubectl get deployment <app> -o jsonpath='{.metadata.annotations.deployment\.kubernetes\.io/revision}'`)
3. on-failure 优先用 snapshot revision 回滚(`kubectl rollout undo deployment/<app> --to-revision=$SNAPSHOT`),fallback `kubectl rollout undo` 回上一版
4. 创建 incident issue,body 含 git sha + workflow run URL + smoke-test annotation 摘录

**Acceptance criteria**:
- [ ] e2e 仓库注入 smoke-test 失败 → deployment 状态最终为上一稳定版(`kubectl describe deployment` 看 revision 历史)
- [ ] `gh issue list --label incident,priority:p1` 出现新 issue,标题含 `P1: Production rollback`
- [ ] issue body 含:故障 image digest、回滚到的 image digest、workflow run URL
- [ ] 重复触发同一回滚两次(模拟 retry):仍只创建一个 issue(用 `gh issue list --search "<commit-sha>"` 去重),或两次都创建但能识别为同一事件
- [ ] auto-rollback 自身失败(如 `gh issue create` 报错)不影响 rollback 主路径
