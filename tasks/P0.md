## P0-1: manifest.yml + manifest-pending.yml + verify-sha-consistency

**Status**: 🟢 Done

**Goal**:建立第三方 action SHA 单一来源(`manifest.yml`)、待验证 SHA 暂存区(`manifest-pending.yml`)与一致性校验 CI job(`verify-sha-consistency.yml`)。

**SPEC ref**:第三章 + 3.1 + 附录 C

**Effort**:M(2 天)

**Dependencies**:无(所有任务的前置)

**Files to create**:
- `manifest.yml`
- `manifest-pending.yml`
- `.github/workflows/verify-sha-consistency.yml`
- `.github/scripts/verify-sha-consistency.sh`
- `tests/scripts/verify-sha-consistency.bats`

**Constraints**(额外):
- `manifest.yml` 顶部注释明确说明:本文件仅放**已验证 SHA**;未验证条目放 `manifest-pending.yml`
- `manifest.yml` 至少包含 `actions/checkout` `step-security/harden-runner` `actions/dependency-review-action` `aquasecurity/trivy-action` `sigstore/cosign-installer` `github/codeql-action` `codecov/codecov-action` `anthropics/claude-code-action`(这些是 P0/P1 立即会用到的)
- `manifest-pending.yml` 占位 SHA 用 `<待验证 SHA>` 字面量,**不**用 `1234...` 假数据
- `verify-sha-consistency.sh` 必须检测:
  1. workflow / action 文件中的 `uses:` SHA 与 `manifest.yml` `deps:` 段一致
  2. 没有 `@v\d+` / `@main` / `@master` 形式引用(本地 `./` 引用除外)
  3. 没有 `manifest-pending.yml` 条目被引用
  4. 没有 SPEC 附录 B.2 已淘汰 action 被引用

**Implementation outline**:
1. 创建 `manifest.yml`,deps 段从 SPEC 第三章已验证清单复制(注意:占位 SHA 必须从 SPEC 转移到 `manifest-pending.yml`)
2. workflows 段列全 14 个 reusable workflow 的 inputs/outputs/secrets 契约(从 SPEC 复制)
3. 编写 `verify-sha-consistency.sh`(伪代码见 SPEC 附录 C),用 `yq` 解析 manifest,`grep -rE` 扫描所有 yaml
4. `verify-sha-consistency.yml` 在 `pull_request` + `push: branches: [main]` 触发,挂在 preflight 之后
5. bats 测试:fixture 一个故意 SHA 错配的 workflow,验证脚本 exit 1

**Acceptance criteria**:
- [ ] `yq '.deps' manifest.yml` 输出非空,所有 value 是 40 位 hex
- [ ] `bash .github/scripts/verify-sha-consistency.sh` 在干净仓库 exit 0
- [ ] 故意把任一 workflow 的 SHA 改 1 位 → 脚本输出 `::error::SHA mismatch` + exit 1
- [ ] 故意把 `uses: foo/bar@v1` 写到任一 workflow → 脚本输出 `::error::non-SHA uses` + exit 1
- [ ] `manifest-pending.yml` 中的 action 在某 workflow 被引用 → 脚本 exit 1 并输出 "Migrate after verification"
- [ ] `bats tests/scripts/verify-sha-consistency.bats` 全绿,覆盖率 ≥ 80%
- [ ] `verify-sha-consistency.yml` 在自身 PR 触发时 self-test 绿

## P0-3: Concurrency Groups 全覆盖

**Status**: 🟡 In Progress(verify-sha-consistency.yml 已应用约束;P1 起其余 workflow 落地时同步嵌入)

**Goal**:为所有 14 个主 workflow 定义 concurrency group,按"PR 类取消旧 run / deploy 类不取消"分类。

**SPEC ref**:5.7

**Effort**:S(0.5 天,但分布在所有 P1+ workflow 创建时同步生效)

**Dependencies**:无独立产物;作为**约束嵌入**P1-8/9/10/11/13、P2-15/16/18、P3-22/23 等所有 workflow 任务。

**Files to modify**(每个 workflow 任务负责自己的 concurrency 块):
- 所有 `.github/workflows/*.yml` 顶层 `concurrency:` 段

**Constraints**(额外):
- 表达式必须考虑 reusable workflow 调用方上下文(SPEC 5.7):`github.ref` 是调用方 ref
- group 名**不**加常量前缀(如 `openCI-`),group 命名空间已天然按仓库隔离
- group 命名约定:

| Workflow | group 表达式 | cancel-in-progress |
| --- | --- | --- |
| `pr.yml` / `docs-build.yml` | `pr-${{ github.event.pull_request.number }}` | `true` |
| `ci.yml` | `ci-${{ github.ref }}` | `false` |
| `stg.yml` | `deploy-stg-${{ github.ref }}` | `false` |
| `prd.yml` | `deploy-prd-${{ github.ref }}` | `false` |
| `claude-harness.yml` | `claude-harness-${{ github.run_id }}` | `false` |
| `health-report.yml` / `security-schedule.yml` | `<workflow-name>` (全局单例) | `false` |
| `release-docker.yml` | `release-${{ github.ref }}` | `false` |
| `docs-deploy.yml` | `pages-${{ github.ref }}` | `false` |
| `stale.yml` / `community.yml` | `<workflow-name>` | `false` |

**Acceptance criteria**(在 P0 阶段验证一次,后续 workflow 任务复用同一约束):
- [ ] 在 P1 第一个 workflow(pr.yml)就绪后,验证:同一 PR 连续 push 两次 → 第一次 run 状态 cancelled,第二次 run 状态 in_progress
- [ ] 在 P1-11 stg.yml 就绪后,验证:对同一 ref 两次手动 dispatch → 两次 run 都 in_progress(不取消)
- [ ] 任何在本任务**之后**新增的 workflow 都遵循上表;违反 → PR review 阻断

## P0-4: Secrets Preflight + preflight-secrets.sh

**Status**: 🟢 Done

**Goal**:每个 workflow 启动时 live-probe 必需 secret,缺失立即 fail(不浪费 runner 时间)。

**SPEC ref**:5.1 + 8.2(消费方 secrets 矩阵)

**Effort**:S(1 天)

**Dependencies**:无独立产物;约束嵌入所有后续 workflow 任务。

**Files to create**:
- `.github/scripts/preflight-secrets.sh`
- `tests/scripts/preflight-secrets.bats`

**Constraints**(额外):
- 接受 `--required "X,Y,Z"` 与 `--optional "A,B"` 两个 CSV 参数
- required 缺失 → `::error title=Missing Secret::SECRET_NAME` + exit 1
- optional 缺失 → `::notice title=Optional Secret Skipped::SECRET_NAME` + exit 0
- **不**打印 secret 值(只打印名字),`set -u` 防止未定义展开为空字符串
- 脚本必须 `< 30 秒` 完成(GC4 时间预算)

**Implementation outline**:
1. shell 解析 `--required` / `--optional`,split CSV
2. 对每个名字读 `${!NAME}`(间接展开),空则按 required/optional 分支
3. 必须用 GitHub Actions annotation 格式输出
4. bats 测试:fixture 设置 / 不设置环境变量,断言 exit code

**Acceptance criteria**:
- [ ] `bash preflight-secrets.sh --required "FOO" --optional ""` 在 `FOO` 未设置时 exit 1
- [ ] 同上,`FOO=bar` 时 exit 0,日志含 `::notice` 而非 `::error`
- [ ] `--optional "BAR"` 在 `BAR` 未设置时 exit 0,日志含 `::notice title=Optional Secret Skipped`
- [ ] 脚本不输出 secret 实际值(grep stdout 不含 `bar`)
- [ ] `bats tests/scripts/preflight-secrets.bats` 全绿
- [ ] 时间测量:`time bash preflight-secrets.sh --required "A,B,C" --optional "D,E,F"` < 1s
- [ ] **后续 workflow 任务**(P1-8 起)`jobs.preflight` 调用本脚本,缺失 required secret 时 workflow 在 30s 内整体失败

## P0-5: graceful-skip 模式

**Status**: 🔴 Not Started(模式定义已在 SPEC §7.3/7.6;待 P1-8 / P2-16 中具体原子 action 实现时落地)

**Goal**:可选安全扫描器(SonarCloud / Snyk / GitGuardian)在 token 缺失时静默跳过,不阻塞 PR。

**SPEC ref**:7.3 / 7.6 / 8.2

**Effort**:S(0.5 天,作为**模式**嵌入 P1-8 / P2-16 的相关原子)

**Dependencies**:P0-4(preflight 区分 required vs optional,本任务实现 optional 的 skip 行为)

**Files to modify**(在对应原子 action 中实现):
- `actions/pr/scan-sonarcloud/action.yml`(P1-8 时实现)
- `actions/pr/scan-snyk/action.yml`(P1-8 时实现)
- `actions/pr/scan-gitguardian/action.yml`(若引入)
- `actions/security/scan-snyk/action.yml`(P2-16 时实现)

**Constraints**(额外):
- 每个原子开头必须有 `Check token` step 检测对应 token 是否设置
- token 缺失 → `echo "skip=true" >> $GITHUB_OUTPUT` + `::notice title=<Tool> Skipped::<TOKEN_NAME> not configured, graceful skip`
- 后续 step `if: steps.check.outputs.skip != 'true'`
- **禁止**将 graceful-skip 应用于 required 类工具(如 dependency-review)。仅限**消费方可选**的 SaaS 集成

**Implementation outline**(单原子模板,后续任务直接套):
```yaml
steps:
  - name: Check token
    id: check
    shell: bash
    run: |
      if [ -z "${{ inputs.<tool>-token }}" ]; then
        echo "skip=true" >> $GITHUB_OUTPUT
        echo "::notice title=<Tool> Skipped::<TOKEN_NAME> not configured"
      fi
  - uses: <vendor-action>@<SHA>
    if: steps.check.outputs.skip != 'true'
    # ...
```

**Acceptance criteria**:
- [ ] 在 P1-8 完成后,新建 PR 不配置 `SONAR_TOKEN`:scan-sonarcloud job 状态 success(skipped),日志含 `::notice title=SonarCloud Skipped`
- [ ] 同上,scan-snyk 不配置 `SNYK_TOKEN` → success(skipped)
- [ ] 配置了 `SONAR_TOKEN` 但**故意**给一个无效 token → SonarCloud 真实失败(因为已不在 skip 路径,正常上抛错误)
- [ ] PR check 总状态:即使所有 graceful-skip 工具都 skip,只要 required jobs(lint/test/scan-deps)都过,PR 可合并

## P0-7: lefthook.yml 本地 hook 模板

**Status**: 🟢 Done

**Goal**:提供本地 git hook 模板,90% lint 问题在 push 前消灭,节省 CI 时间。**消费方可选启用**。

**SPEC ref**:附录 A 第 7 行(aicert #20)

**Effort**:S(0.5 天)

**Dependencies**:无

**Files to create**:
- `lefthook.yml`(项目根)
- `docs/setup-lefthook.md`(消费方启用指南)

**Constraints**(额外):
- 本任务交付的是**模板**,OpenCI 自身可启用,但消费方默认 opt-out
- pre-commit 必须含 5 个 guard:
  - `guard-no-main-commit`(禁止直推 main)
  - `guard-dotenv`(staged 含 `.env` → fail)
  - `guard-large-files`(staged 文件 > 512KB → fail)
  - 语言无关 lint:按 detected language 跑 ruff/eslint/golangci-lint
  - `guard-secrets`(可选,gitleaks 本地扫描)
- commit-msg 必须含 conventional-commit 校验(正则 `^(feat|fix|refactor|docs|test|chore|perf|ci|build|style)(\(.+\))?: .{1,}`)
- pre-push 跑全量 lint(允许 verbose),不跑 test(test 留给 CI)

**Implementation outline**:
1. 编写 `lefthook.yml`(模板见 SPEC 16.5.1 已删除但参考 aicert 原始 lefthook.yml)
2. 文档说明 `brew install lefthook && lefthook install` 两步启用
3. 提供 `--no-verify` skip 但在 `setup-lefthook.md` 里**强烈反对**

**Acceptance criteria**:
- [ ] `lefthook install` 后 `cat .git/hooks/pre-commit` 含 lefthook stub
- [ ] 直接 `git commit` 到 main 分支 → pre-commit 拦截 + 提示
- [ ] staged `.env` 文件 → pre-commit 拦截
- [ ] commit message `add foo` → commit-msg 拦截,提示 conventional-commit 格式
- [ ] commit message `feat: add foo` → 通过
- [ ] `setup-lefthook.md` 含完整启用 + 卸载 + bypass(紧急情况)说明
