## P3-22: community.yml + stale.yml + issue.yml + issue-comment.yml

**Status**: 🔴 Not Started

**Goal**:实现 issue 生命周期管理 4 个 workflow:新贡献者欢迎 + slash command + AI triage + 过期清理。

**SPEC ref**:第六章

**Effort**:L(5 天,涉及 issue templates 设计 + AI prompt 调优)

**Dependencies**:P1-9(claude-harness)、P0-6(CODEOWNERS)

**Files to create**:
- `.github/workflows/issue.yml`
- `.github/workflows/issue-comment.yml`
- `.github/workflows/community.yml`
- `.github/workflows/stale.yml`
- `.github/ISSUE_TEMPLATE/bug-report.yml`
- `.github/ISSUE_TEMPLATE/feature-request.yml`
- `.github/ISSUE_TEMPLATE/question.yml`
- `.github/ISSUE_TEMPLATE/security-report.yml`(提示走私下渠道)
- `.github/ISSUE_TEMPLATE/config.yml`(`blank_issues_enabled: false` + 外链)
- `actions/issue/auto-label/action.yml`(原子,基于 form 字段打 label)
- `actions/issue/ai-triage/action.yml`(原子,内部 `_common/claude-harness`)
- `actions/issue/detect-duplicates/action.yml`(原子)
- `actions/issue/welcome-contributor/action.yml`(原子)
- `actions/issue/auto-assign/action.yml`(原子,基于 CODEOWNERS + area label)
- `actions/issue/parse-command/action.yml`(原子)
- `actions/issue/execute-command/action.yml`(原子)
- `actions/issue/validate-form/action.yml`(原子)
- `actions/community/stale-mark/action.yml` `stale-close/action.yml` `lock-resolved/action.yml`
- `prompts/issue/triage.md`
- `tests/actions/auto-label.bats` `ai-triage.bats` `detect-duplicates.bats` `parse-command.bats`

**Constraints**(额外):
- `issue.yml` 触发:`issues: types: [opened]`
- `issue-comment.yml` 触发:`issue_comment: types: [created]`,`if: contains(github.event.comment.body, '/')` 过滤(减少无效 run)
- `community.yml` 触发:`pull_request: [opened]` + `issues: [opened]` + `issue_comment: [created]`(同 if 过滤)
- `stale.yml` 触发:`schedule: cron: '0 2 * * *'`(每天 UTC 02:00)+ `workflow_call`
- AI triage **必须**:
  - context via `jq -n --arg`(防 prompt injection,GC2/GC3)
  - prompt 指令 AI 输出 JSON `{priority, reasoning, suggested_assignee, is_security}`
  - shell 用 `jq -r` 解析,据此调 `gh issue edit --add-label`
  - 安全相关 issue 自动加 `security` + `private-discuss` label
- slash command 权限判定:`github.event.comment.author_association ∈ {OWNER, MEMBER, COLLABORATOR}` 视为授权;`/help` 任何人可用
- 支持的命令(精确实现 SPEC 6.3 表):`/assign @user` `/unassign` `/label name` `/unlabel name` `/priority p1` `/close` `/reopen` `/duplicate #N` `/needs-info` `/triage` `/help`
- stale workflow 用 `stale-org/stale`(原 `actions/stale` 已归档,见附录 B.2)
- stale 豁免 label:issue `pinned,security,help-wanted,priority:p0,priority:p1`;PR `pinned,security,wip`
- stale 默认参数:issue stale 60d / close 14d;PR stale 30d / close 7d
- lock-resolved:`dessant/lock-threads`,关闭 30 天后锁评论

**Implementation outline**:
1. 先建 5 个 issue template(bug-report 字段严格按 SPEC 6.1)
2. `auto-label` 原子:从 issue body 用 `gh api` 解析 form 字段,映射到 label(`area:frontend` 等)
3. `detect-duplicates`:`gh issue list --search "<title 关键词去 stopword>"` 找 top 5,有则评论 + 打 `possible-duplicate` label
4. `ai-triage`:严格按 SPEC 6.2 安全模板,jq -n 构造 context
5. `welcome-contributor`:`if: github.event.issue.author_association == 'FIRST_TIME_CONTRIBUTOR'`,gh issue comment
6. `auto-assign`:从 CODEOWNERS + area label 推算 assignee,`gh issue edit --add-assignee`
7. `parse-command` / `execute-command`:从 comment body regex 提取命令,按白名单分支
8. `community.yml` 串接 PR/issue/comment 三种触发,按事件类型分支
9. `stale.yml` 包装 stale-org/stale 与 dessant/lock-threads,inputs 暴露 4 个 days 参数

**Acceptance criteria**:
- [ ] 用模板创建 bug issue → 自动打 `type:bug` `area:<选的>` `severity:<选的>` `priority:<AI 决定>` 至少 4 个 label
- [ ] 创建 issue 含 "API key" 字样 → AI triage 标记 `security` + `private-discuss`
- [ ] 第一次提 issue 的用户 → welcome-contributor 评论,含贡献指南链接
- [ ] 创建相似标题 issue → detect-duplicates 评论提示 + 打 `possible-duplicate`
- [ ] OWNER 评论 `/assign @other` → 该 issue assignee 变 @other
- [ ] CONTRIBUTOR 评论 `/close` → 不生效(权限不够),日志 `::notice title=Unauthorized Command`
- [ ] 任何人评论 `/help` → bot 回复支持的命令清单
- [ ] 60 天无活动 issue → stale 标记;再 14 天 → 自动关闭
- [ ] 带 `pinned` label 的 issue → 永不 stale
- [ ] 关闭超过 30 天的 issue → 评论被锁

## P3-23: docs-build.yml + docs-deploy.yml + release-docker.yml

**Status**: 🔴 Not Started

**Goal**:文档构建 / 文档部署到 GitHub Pages / Docker 镜像 tag 触发发布,共 3 个工具型 workflow。

**SPEC ref**:第十九章

**Effort**:M(2.5 天)

**Dependencies**:P0-1, P0-3, P0-4

**Files to create**:
- `.github/workflows/docs-build.yml`
- `.github/workflows/docs-deploy.yml`
- `.github/workflows/release-docker.yml`
- `tests/scripts/markdown-link-check-fixtures/`(测 docs-build 链接校验)

**Constraints**(额外):
- `docs-build.yml`:触发 `pull_request` paths `docs/**` + `*.md`;timeout 5 min;step `markdown-link-check docs/**/*.md` 验证链接
- `docs-deploy.yml`:触发 `push: branches: [main]` paths `docs/**` + `*.md` + `workflow_dispatch`;permissions `contents:read pages:write id-token:write`;environment `github-pages`;step `actions/upload-pages-artifact` + `actions/deploy-pages`
- `release-docker.yml`:触发 `push: tags: ['v*']` + `workflow_dispatch` (input `tag`);permissions `contents:read packages:write`;timeout 20 min;steps `checkout` → `setup-buildx-action` → `login-action` (registry=ghcr.io, password=GITHUB_TOKEN) → `metadata-action` (tags semver `{{version}}` / `{{major}}.{{minor}}` / `type=sha`) → `build-push-action` (push: true, cache-from/to: type=gha,mode=max)
- 文档构建命令由消费方决定(MkDocs `mkdocs build --strict` / Docusaurus `npm run build`),通过 input `build-cmd` 传入,默认尝试 `mkdocs build --strict`
- `docs-deploy` 与 `release-docker` 不取消进行中的 run(`cancel-in-progress: false`)
- `docs-build` 取消旧 run(`cancel-in-progress: true`)

**Implementation outline**:
1. docs-build:checkout → 跑 `inputs.build-cmd` → `npx markdown-link-check`
2. docs-deploy:同 build + upload-pages-artifact + deploy-pages
3. release-docker:5-step Docker build push 标准序列

**Acceptance criteria**:
- [ ] 修改 `docs/SPEC.md` 提 PR → docs-build 触发,success
- [ ] 在 docs/ 加一个无效链接 `[bad](http://does-not-exist.example.invalid)` → docs-build fail
- [ ] merge 到 main(改 docs/) → docs-deploy 跑,GitHub Pages URL 可访问
- [ ] `git tag v1.0.0 && git push --tags` → release-docker 触发,GHCR 出现 `:1.0.0` `:1.0` `:sha-abc1234` 三个 tag
- [ ] release-docker run buildx cache 命中(同一 commit 第二次跑用时 < 30s,首次 ~5min)

## P3-25: Dependabot Auto-Merge(patch 自动合并)

**Status**: 🔴 Not Started

**Goal**:Dependabot / Renovate 提的 patch 级 PR 自动合并,major / runtime-minor 仍人工审查。

**SPEC ref**:aicert #22

**Effort**:S(1 天)

**Dependencies**:P0-1(SHA pinning,Renovate 是首选)

**Files to create**:
- `.github/renovate.json`(主配置,见 SPEC 17.4)
- `.github/workflows/dep-auto-merge.yml`(订阅 Renovate PR + 自动 enable auto-merge)
- `tests/scripts/renovate-config-validate.bats`(用 `renovate-config-validator` CLI 校验)

**Constraints**(额外):
- 只对 **patch + pin + digest** 自动合并,**不**对 major / runtime-minor 自动合并
- 自动合并必须 wait for **所有 required checks** 通过(GitHub native auto-merge)
- 修改的依赖如果是 `manifest.yml` 的 SHA 更新 → 还需 P0-1 的 verify-sha-consistency 通过(本来就是 required check)
- 配置必须用 Renovate JSON Schema(顶部 `$schema`),保证 IDE 提示
- dependabot.yml 作为备选,但**不**启用 auto-merge(Dependabot 不支持 SHA pinning,潜在风险)

**Implementation outline**:
1. 写 `.github/renovate.json`,严格按 SPEC 17.4 推荐配置
2. dep-auto-merge.yml 订阅 PR 事件,过滤 author=renovate[bot] + label=patch,调 `gh pr merge --auto --squash`
3. bats 测试用 `npx --package renovate -- renovate-config-validator` 校验配置合法性

**Acceptance criteria**:
- [ ] Renovate 提一个 patch PR(模拟:手动改一个依赖 patch 版本) → labels 含 `deps`,几分钟内 PR 自动设 auto-merge
- [ ] 所有 required check 绿后 → PR 自动 squash merge
- [ ] Renovate 提 major PR → label 含 `breaking-change`,**不**自动 merge,等人工审批
- [ ] 故意提 PR 改 manifest.yml SHA 但不更新引用 → verify-sha-consistency fail → auto-merge 不发生(因为 required check 没通过)

**Acceptance criteria**:
- [ ] OpenCI 仓库 `docs/ENV_MATRIX.md` 渲染良好,含 5+ 示例变量
