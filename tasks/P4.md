## P4-28: Issue → Branch 自动化(Linear webhook 驱动)

**Status**: 🟢 Done(linear-bridge / linear-comment 原子 + repository_dispatch workflow + Cloudflare Worker 桥接示例 + setup 文档;6 个 derive-branch bats 全绿)

**Goal**:Linear 中 issue 状态变 `In Progress` 时,自动在 GitHub 仓库创建 `feat/<linear-id>-<slug>` 分支并回写 Linear 评论(关联到分支)。

**SPEC ref**:aicert #1

**Effort**:M(3 天,涉及 Linear webhook 桥接)

**Dependencies**:无独立 OpenCI 依赖,但消费方需有 Linear 账号 + workspace

**Files to create**:
- `.github/workflows/issue-branch-from-linear.yml`(触发 `repository_dispatch: types: [linear-issue-started]`)
- `actions/integrations/linear-bridge/action.yml`(原子,处理 Linear webhook payload + gh API 创建分支)
- `actions/integrations/linear-comment/action.yml`(原子,回写 Linear comment)
- `docs/setup-linear-webhook.md`(消费方在 Linear 配 webhook 指南)
- `tests/actions/linear-bridge.bats`

**Constraints**(额外):
- Linear webhook 不能直接调 GitHub workflow,中间用一个**桥接函数**(消费方部署 cloudflare worker / aws lambda),把 webhook 转成 `gh api repos/.../dispatches`
- 桥接代码示例放在 `examples/linear-webhook-bridge/`(node + cloudflare-worker 各一份)
- 分支名规则:`<type>/<linear-id-lowercase>-<slug>`,例:`feat/aic-123-add-login`
- type 由 Linear issue label 决定:label 含 `bug` → `fix/`;`feature` → `feat/`;其他 → `chore/`
- slug 由 issue title 经 `tr -c '[:alnum:]' '-' | tr -s '-' | tr '[:upper:]' '[:lower:]'` 处理
- 创建分支后,**自动**回写 Linear 评论:"Branch created: <github-branch-url>"
- 如果 issue 重复触发(状态来回切),已存在分支 → skip 创建 + 回写 "Branch already exists"

**Implementation outline**:
1. 桥接函数(示例代码):验证 Linear webhook signature → 提取 issue id/title/labels → POST `gh api`
2. `linear-bridge` action:从 `github.event.client_payload` 读 issue 信息,`gh api repos/.../git/refs` 创建分支
3. `linear-comment` action:`curl POST https://api.linear.app/graphql` mutation `commentCreate`
4. 文档说明 webhook 配置 + 桥接函数部署

**Acceptance criteria**:
- [ ] Linear 测试 workspace 创建 issue,标记 `In Progress` → 5 分钟内 GitHub 仓库出现 `feat/aic-XXX-<slug>` 分支
- [ ] Linear issue 评论列表新增一条 "Branch created: ..." 链接到 GitHub 分支
- [ ] 同一 issue 重复触发 → 不创建重复分支,Linear 评论 "Branch already exists"
- [ ] webhook signature 校验失败 → 桥接函数 return 401,不调 GitHub
- [ ] 分支命名严格符合 `<type>/<id>-<slug>` 规则,无空格 / 大写

## P4-29: Agent 反馈闭环(copilot-swe-agent / codex/* 自动 @-mention)

**Status**: 🟢 Done(identify-pr-agent + summarize-failure 原子 + pr-agent-feedback workflow,sticky comment 通过 sha-marker 去重;6 个 identify-pr-agent bats 全绿)

**Goal**:CI 失败时自动识别 PR 作者是 AI agent(`copilot-swe-agent[bot]` / 分支名 `codex/` 前缀),@-mention 该 agent 附上失败日志摘要让其修复。

**SPEC ref**:aicert #5

**Effort**:M(2.5 天)

**Dependencies**:P1-8(pr.yml)、P1-9(claude-harness 用于失败日志摘要)、P2-19(workflow_run 聚合)

**Files to create**:
- `.github/workflows/pr-agent-feedback.yml`(订阅 workflow_run,识别 agent + 评论 mention)
- `actions/_common/identify-pr-agent/action.yml`(原子,识别 PR 作者类型)
- `actions/_common/summarize-failure/action.yml`(原子,从 workflow logs 抽取失败摘要,内部 claude-harness)
- `tests/actions/identify-pr-agent.bats`

**Constraints**(额外):
- 识别规则:
  - `github.event.pull_request.user.login == 'copilot-swe-agent[bot]'` → copilot
  - `github.head_ref` 以 `codex/` 开头 → codex
  - 分支名匹配 `dev-agent/<id>` → 自定义 OpenCI agent(预留扩展点)
- 触发只在 `pr.yml` / `ci.yml` `workflow_run.conclusion == 'failure'`(success 不触发)
- mention 评论格式:
  ```
  @<agent> CI failed on this PR.
  Failure summary: <claude-summarized-1-line>
  Failed jobs: <list>
  Run: <link-to-workflow-run>
  ```
- 同一 PR 同一 commit-sha 失败 → 只评论 1 次(用 `<!-- agent-feedback:<sha> -->` marker 去重)
- 不识别为 agent → workflow exit 0,无操作(不污染普通 PR)

**Implementation outline**:
1. `identify-pr-agent`:读 `github.event.pull_request.user.login` + `github.head_ref`,emit step output `agent-type`(copilot/codex/openci/none)
2. `summarize-failure`:gh api 拉 workflow run logs,分批喂给 claude-harness 总结(限 1 行)
3. `pr-agent-feedback.yml`:订阅 workflow_run,if agent-type != none + conclusion == failure → 评论
4. 评论用 `<!-- agent-feedback:<sha> -->` marker 防重复

**Acceptance criteria**:
- [ ] 模拟 PR by `copilot-swe-agent[bot]`(本地用 act + mock event payload)CI 失败 → PR 出现一条 `@copilot-swe-agent` 评论
- [ ] 评论含失败 job 名 + 1 行摘要 + run URL
- [ ] 普通用户 PR CI 失败 → 无 agent 评论
- [ ] 同 PR 同 sha 失败两次 → 只有一条评论(marker 去重)
- [ ] PR push 新 commit 后再失败 → 出现新评论(不同 sha)

## P4-31: Canary Watch / Verify Fix / Terraform Drift

**Status**: 🟢 Done(canary-watch 3σ math + verify-fix ref parsing 完整 + bats 全绿;prd-canary-watch.yml 已接入 Sentry Stats v2 实际 API 拉取 current + 7-day history rates;terraform-drift 已接入 hashicorp/setup-terraform@5e8dbf3c (v4.0.0 SHA 已加入 manifest.yml),实际 plan -refresh-only -detailed-exitcode 逻辑完整;terraform-drift.bats 10 个测试全绿)

**Goal**:三个部署后监控 workflow:canary 监控(15 分钟周期 Sentry 3σ 偏离 → 自动回滚)、修复验证(prd 部署后对照 PR 的 `Fixes #N` 验证 Sentry 错误是否真消失)、Terraform 漂移检测(advisory)。

**SPEC ref**:aicert #12-#14

**Effort**:L(5 天)

**Dependencies**:P1-13/14(prd.yml 与回滚机制),P2-15(check-error-rate 公用 Sentry 客户端逻辑)

**Files to create**:
- `.github/workflows/prd-canary-watch.yml`(`schedule: cron: '*/15 * * * *'`,部署后 30 分钟内监控)
- `.github/workflows/prd-verify-fix.yml`(prd.yml 完成后 `workflow_run` 触发)
- `.github/workflows/prd-terraform-drift.yml`(prd.yml 之前一步,advisory)
- `actions/prd/canary-watch/action.yml`(原子,3σ 检测 + 回滚触发)
- `actions/prd/verify-fix/action.yml`(原子,PR body parse + Sentry resolved check)
- `actions/prd/terraform-drift/action.yml`(原子,`terraform plan` only-read)
- `tests/actions/canary-watch.bats` `verify-fix.bats`

**Constraints**(额外):
- canary-watch:
  - 仅在最近 30 分钟内有 prd 部署时跑(读 PRD 部署 marker)
  - 3σ 检测:取过去 7 天的同时段错误率 mean+3*stddev 作为基线
  - 触发回滚:调 `actions/prd/auto-rollback`(P1-14)+ 创建 P1 issue
- verify-fix:
  - 解析 prd.yml 关联 PR 的 body,提取 `Fixes #N` / `Closes #N` 编号
  - 对每个 issue,Sentry API 查询该 issue 关联的 error 是否在过去 1h 仍出现
  - 仍出现 → 评论 PR + 重开 GitHub issue;消失 → 评论 PR confirmed fixed
- terraform-drift:
  - 跑 `terraform plan -detailed-exitcode -refresh-only`,exit code 2 = drift
  - **advisory 模式**:即使有 drift 也只发 issue 不阻断部署
  - 仅消费方有 `infrastructure/` 目录时跑

**Implementation outline**:
1. canary-watch:用 P2-15 的 Sentry Stats API,加 std dev 计算
2. verify-fix:用 GitHub API 找 prd 关联 PR,grep `(?i)(fixes|closes)\s+#(\d+)` regex
3. terraform-drift:`hashicorp/setup-terraform@<SHA>` + `terraform plan`,捕获 exit code 2
4. 三个 workflow 各自串接

**Acceptance criteria**:
- [ ] 部署后人为制造高错误率 → canary-watch 在 15 分钟内触发,自动回滚 + 开 P1 issue
- [ ] PR 含 `Fixes #100` merge 到 prd → 1 小时后 verify-fix 跑;Sentry 中 #100 关联的 error 已消失 → PR 评论"Confirmed fixed: <Sentry link>";仍存在 → 评论 + 重开 issue #100
- [ ] terraform plan 在 stg 环境注入手动改动 → terraform-drift 创建 issue 标记 `infra-drift,priority:p2`,但 prd.yml 仍可继续(advisory)
- [ ] 无 `infrastructure/` 目录的消费方 → terraform-drift skip,annotation `::notice title=Terraform Drift Skipped::no infrastructure/ dir`
